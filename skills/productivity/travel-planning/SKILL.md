# 旅行规划

## 收集约束

先从对话和记忆提取：

- 出发地、目的地、日期和可用天数
- 同行者、预算、交通方式和住宿偏好
- 行程节奏、饮食兴趣、必去与明确不想去的内容
- 行动不便、过敏、签证或其他硬约束

只询问会显著改变行程的信息。用户允许自由发挥时，不要用长问卷拖住任务。

## 研究与规划

1. 使用 `maps-and-places` 处理地点、区域、距离和路线。
2. 使用 `web-research` 核实开放时间、预约规则、交通调整、节假日和价格等时效信息。
3. 使用 `calculator` 汇总预算或时间，不制造虚假精度。
4. 按地理邻近分组，先排交通骨架，再放景点、餐饮和休息。
5. 每半天设置一个主要目标，保留排队、换乘、天气和临时改变的余量。
6. 多日复杂规划使用 `update_plan` 管理研究与交付进度。

## 质量要求

- 区分已核实信息、估算和个人推荐。
- 不把“热门”当作适合用户；优先使用已知偏好。
- 餐饮计划考虑营业时段、排队、距离和连续重口味疲劳。
- 自驾计划考虑停车、取还车、充电或加油以及驾驶疲劳。
- 不同城市之间明确交通时间，不把移动日排成完整游玩日。
- 给出雨天、闭馆或体力不足时的替代方案。

## 输出

按天展示：区域、主要活动、交通、餐饮建议、预计花费和备选项。附一段预订/出发前检查清单。用户要求保存时使用 `write_file` 生成 Markdown；否则直接在对话中交付。

## 边界

- 不声称已经预订、购票或联系商家，除非真实工具结果证明完成。
- 票价、营业时间和实时交通注明查询日期。
- 精确住址、证件和行程定位默认不写入长期记忆。
