# 系统排错

随机修 bug 浪费时间、引入新问题。症状修补只是自欺欺人。

**核心原则:** 找到根因之前不写修复代码。修症状 = 失败。

```
没有根因调查 → 不能提出修复方案
```

## 适用场景

任何技术问题：
- 测试失败、生产 bug
- 意外行为、性能问题
- 构建失败、集成问题

**特别必须在以下情况使用:**
- 时间紧（紧急情况诱惑乱猜）
- "很明显"应该这样修
- 已经试了好几次
- 上次修复没生效
- 你有不完全懂的地方

## 四阶段流程

### 阶段 1 — 根因调查

**修之前必须做:**

1. **仔细读错误信息**
   - 完整读堆栈，不跳过
   - 记下文件路径、行号、错误码
   - 用 read_file 看源码，用 search_files 搜错误字符串

2. **稳定复现**
   - 每次都能触发吗？精确步骤是什么？
   - 不能复现 → 收集更多数据，别猜

```bash
# 跑失败的测试
pytest tests/test_module.py::test_name -v --tb=long
```

3. **检查近期改动**
```bash
git log --oneline -10
git diff
```

4. **追踪数据流**
   - 坏值从哪来？
   - 谁传的坏值？
   - 一直追到源头

5. **多组件系统诊断**
   - 每个组件边界：记录输入 → 记录输出 → 确认环境变量 → 检查状态
   - 跑一次收集证据 → 分析 → 定位

**阶段 1 完成标志:** 理解了什么出错、为什么出错。

### 阶段 2 — 模式分析

1. **找能工作的例子** — 代码库里有没有类似的正常代码？
2. **对比差异** — 正常的和出错的有什么不同？每条差异都列出来
3. **理解依赖** — 这个组件需要什么？什么配置和假设？

### 阶段 3 — 假设和验证

1. **形成单一假设** — 明确说"我认为 X 是根因，因为 Y"
2. **最小验证** — 做最小改动验证假设，一次只改一个变量
3. **如果没对** — 形成新假设，不堆叠修复

当不知道时直接说"我不懂 X"，不假装。

### 阶段 4 — 实施修复

1. **先写复现测试** — 最简单的复现，自动化测试
2. **单一修复** — 改根因，一个改动
3. **验证修复**
```bash
pytest tests/test_module.py::test_regression -v
pytest tests/ -q  # 全量
```

4. **三条规则** — 如果试了 3 次还不行:
   - 停下来
   - 不是修错了，是架构问题
   - 和用户讨论是否重构

## 红旗 — 停下来回到阶段 1

- "先快速修一下，之后再查"
- "试试改 X 看行不行"
- "多个改动一起跑"
- "应该是 X，让我修一下"
- "不完全懂但这样应该行"
- "再试一次"（已经试了 2+）
- 提出修复方案但没追踪数据流
- 每次都揭示新问题

**以上任何一条 = 回到阶段 1。**

## 快速参考

| 阶段 | 活动 | 成功标志 |
|------|------|---------|
| 1. 根因 | 读错误、复现、查改动、追踪数据 | 理解什么和为什么 |
| 2. 模式 | 找能工作的例子、对比差异 | 知道区别在哪 |
| 3. 假设 | 形成理论、最小验证 | 确认或推翻 |
| 4. 实施 | 写回归测试、修复根因、验证 | bug 解决、全量测试通过 |

## 真实影响

- 系统排查: 15-30 分钟修好
- 随机修: 2-3 小时来回折腾
- 一次修对率: 95% vs 40%
