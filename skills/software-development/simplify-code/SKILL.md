# 简化代码

审查近期代码改动，从三个维度逐个检查并应用优化。

**核心原则:** 三个窄维度的系统性审查胜过一次笼统的 review。

## 适用场景

- 用户说"简化"、"精简"、"清理代码"
- 完成功能后回头看
- 跳过: 纯配置、纯文档

## 流程

### 第一步 — 获取改动

```bash
# 默认: 未提交的改动
git diff

# 如果为空
git diff HEAD

# 特定范围
git diff --staged              # 暂存区
git diff HEAD~1                # 最近一次提交
git diff -- src/foo.py         # 特定文件
```

如果没有 git 仓库或没有改动，用户指定了文件就用文件，否则说没东西可审查。

### 第二步 — 三维度审查

逐个检查改动的代码：

**维度 1 — 代码复用**
- 新代码是否重复了已有功能？
- 用 search_files 搜代码库，找现成的工具函数、常量、模式
- 手写的字符串/路径处理、自定义类型守卫、重复的解析逻辑

**维度 2 — 代码质量**
- 冗余状态（可推导的值）
- 参数膨胀（该重构的函数）
- 复制粘贴变体（应提取抽象）
- 抽象泄漏（暴露内部实现）
- 字符串类型代码（该用常量/enum）

**维度 3 — 效率**
- 重复计算、重复文件读取、重复 API 调用
- N+1 访问模式
- 启动或热路径上的重操作
- 不必要的预检
- 一次性加载全文件

### 第三步 — 应用优化

1. 合并三个维度的发现
2. 去重和丢弃误报
3. 应用优化，一次改一个
4. 跑相关测试确保没破坏
5. 总结：改了哪些、跳过了哪些、为什么

## 输出格式

```
## 简化结果

### 应用了
- **文件:行号** — 优化内容

### 跳过了
- **发现** — 原因

### ✅ 测试状态
- 全部通过 / 未跑测试
```

## 注意事项

- 不是全面重构，聚焦在近期改动范围
- 不添加新功能
- 风格改动不单独提，除非影响可读性
- 有疑问时保留原样
